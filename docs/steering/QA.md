# Microflow QA Standards

> Generated by `steer-qa`. Refine as the test strategy evolves — do not regenerate from scratch.

Last updated: 2026-05-16

## Test Strategy

The Rust runtime is the only layer under automated test today. TypeScript apps (web, server, plugins, docs) are gated by `check-types`, `oxlint`, and `oxfmt` — no JS test framework is configured.

| Level | Layer | Tool | Purpose |
| --- | --- | --- | --- |
| Integration | Rust runtime (`apps/web/src-tauri`) | `cargo test --lib --tests` | Verify FlowExecutor edge routing, component lifecycle, board state, MQTT reconnect, value sync, event forwarding |
| Property | Rust runtime (where exists) | `proptest` | Random-input checks (dev-dep available; reach for it on routing / value-coercion code) |
| Benchmark | Rust runtime | `criterion` | `event_routing` bench — performance regressions on the hot path |
| Type / lint | TS workspace | `tsc --noEmit`, `oxlint`, `oxfmt` | No type, lint, or format drift |
| Static analysis | Rust | `cargo clippy --all-targets -- -D warnings -W clippy::pedantic` | Pedantic clippy is a CI gate |
| Cross-platform build | Tauri matrix | `tauri-action` | macOS Intel + ARM, Windows MSVC, Linux x86_64 must all build |

End-to-end tests against a real Firmata board are **not in CI** — they are a manual gate (see Definition of Done).

## Coverage Requirements

No coverage tool is wired up today, so the rule is **risk-based, not percentage-based**:

- The critical paths must have integration tests: event routing, board state / IO loop, wiring registry, component lifecycle, MQTT reconnect, value sync. (Already covered by `apps/web/src-tauri/tests/*.rs`.)
- Every PR adds at least one test that **would have caught the change breaking** — bug fix, refactor, or new feature. The test grows with the code.
- When a coverage tool is added (likely `cargo-llvm-cov`), the threshold will be set against the snapshot at that moment.

## Test Patterns

- **Location:** Rust integration tests in `apps/web/src-tauri/tests/<name>.rs`. Shared helpers in `apps/web/src-tauri/tests/common/` (`mock_board.rs`, `mock_component.rs`, `mod.rs`).
- **Naming:** one file per behavior cluster (`event_routing.rs`, `board_state.rs`, `mqtt_reconnect.rs`, …). Test names describe observable behavior, not method names.
- **Mocks:** stay at the hardware seam. Use `MockBoard` instead of touching a real serial port; use `MockComponent` instead of constructing real impls when the test is about routing rather than component logic. Do not mock internal runtime types (`FlowExecutor`, `WiringRegistry`, etc.) — exercise them directly.
- **Errors:** assert against typed `RuntimeError` variants, never on string content.
- **Async:** use `tokio` test runtime; never `block_on` from a sync test.

## Gherkin → Test Mapping

When a Task issue carries Gherkin scenarios:

- Each scenario maps to at least one Rust integration test.
- Test name references the scenario name so a failure points back to the ticket.
- `Given` → setup (build a flow + mocks). `When` → action (`FlowExecutor::dispatch`, `BoardHandle::digital_write`, etc.). `Then` → assertions on the mock or on runtime state.
- Scenarios describe observable runtime behavior, not implementation.

## Commands

```bash
# Lint + format (CI gate, TS)
bun run check

# Type check (CI gate, TS)
bun run check-types

# Clippy pedantic (CI gate, Rust)
cd apps/web/src-tauri && cargo clippy --all-targets -- -D warnings -W clippy::pedantic

# Run all Rust tests (CI gate)
cd apps/web/src-tauri && cargo test --lib --tests

# Run a single integration test file
cd apps/web/src-tauri && cargo test --test event_routing

# Run a single test by name
cd apps/web/src-tauri && cargo test --test event_routing -- routes_fan_out

# Benchmarks (not in CI)
cd apps/web/src-tauri && cargo bench --bench event_routing

# Cross-platform Tauri build (matches build.yml matrix) — local equivalent
cd apps/web && bun run desktop:build
```

## Definition of Done

A Task is ready to merge when **all** of the following are true:

- [ ] `bun run check` passes (oxlint + oxfmt)
- [ ] `bun run check-types` passes
- [ ] `cargo clippy --all-targets -- -D warnings -W clippy::pedantic` passes
- [ ] `cargo test --lib --tests` passes
- [ ] Cross-platform Tauri build passes (build.yml matrix: macOS Intel / ARM, Windows MSVC, Linux x86_64)
- [ ] If `node-components.json` changed: `bun run codegen` was run and both `_REGISTRY.ts` and the Rust registry regenerated
- [ ] If the Task has Gherkin scenarios: each scenario has a corresponding Rust integration test
- [ ] If the change touches hardware code (`BoardHandle`, `Board IO Loop`, `BoardConnection`, `flasher`, or a `HardwareComponent` impl): manual smoke on a real Firmata board before merge — mocks are not a substitute when the wire matters
- [ ] If domain terms changed or new terms were introduced: `CONTEXT.md` updated
- [ ] If an architectural decision was made: an ADR is added under `docs/adr/NNNN-*.md`
- [ ] Regression test added for any bug fix

## Test Environments

| Environment | Purpose | Access |
| --- | --- | --- |
| Local — no hardware | Run all Rust integration tests with mocks | `cd apps/web/src-tauri && cargo test` |
| Local — with Firmata board | Manual smoke for hardware-touching changes | Plug Firmata-flashed board into USB; run `bun run desktop:dev` |
| Local — DB-backed packages | Exercise `@microflow/db`, `@microflow/collab`, `@microflow/auth` | `bun run db:start` (docker compose) then run the relevant app |
| CI — Rust | Clippy + test gate on PR to `main` | `.github/workflows/rust.yml` |
| CI — Build | Cross-platform Tauri build on PR to `main` | `.github/workflows/build.yml` |
| CI — Release | Signed release artifacts | `.github/workflows/release.yml` on `v*` tag |

## Known Flaky Areas

- **MQTT reconnect tests under load.** `mqtt_reconnect.rs` covers the happy path; edge-case timing under load can vary across runs. When a failure in this file appears once and disappears on rerun, file a ticket rather than ignoring — repeated single-shot failures are a real signal.

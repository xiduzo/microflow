## Product Purpose

Microflow lets people give microcontrollers behavior by drawing a flow on a canvas instead of writing C++. A Firmata-flashed board is the runtime target; the flow editor drives pins, sensors, and protocols in real time over a local serial connection. Designers using Figma or Penpot can plug their work into the same flows through an MQTT bridge, and a real-time collaborative canvas lets multiple people prototype together on one board.

## Target Users

| Actor | Context | Primary goals |
| --- | --- | --- |
| Maker / hardware prototyper | Hobby bench, hackerspace, agency prototype | Wire up sensors and actuators and iterate on behavior without firmware code |
| Designer | Figma / Penpot design files driving physical mock-ups | Connect a design artefact to real hardware via MQTT, no firmware engineer needed |
| Educator | Physical-computing classroom or workshop | Teach interaction concepts without spending the first week on Arduino syntax |
| Student | Self-led learning of physical computing | Build a working circuit + behavior on day one; understand by seeing flow execute |

## Core Principles

1. **Low floor, high ceiling** — A beginner with zero hardware knowledge gets a flow running in minutes. A professional building interactive installations gets the depth they need for serious prototypes. Neither group is the secondary citizen.
2. **Local-first, desktop-native** — Microflow Studio runs on the maker's machine and talks to the board over the local serial port. The browser is not a target; the cloud is not a dependency for the core loop.
3. **Visual flow over text code** — The canvas is the program. Dropping into a code editor is never the price of admission for common behavior.
4. **Single source of truth for components** — The Component Catalog (`apps/web/node-components.json`) drives both the UI registry and the Rust runtime registry. UI and runtime cannot drift.
5. **Domain language is enforced** — Code, docs, and conversations use the terms in `CONTEXT.md` (Component, Port, Wiring, BoardHandle, Hardware Callback, …). Generic words like "node" or "handler" are not substitutes.

## Strategic Goals

- [ ] First flow running on a board in under 10 minutes for an absolute beginner with no hardware background
- [ ] Trusted by professional interactive-installation creators for production prototypes
- [ ] Component library covers the common sensor/actuator surface (digital, analog, I2C, stepper, servo) and the common protocols (MQTT, LLM, Firmata)
- [ ] Self-sustaining via Polar subscriptions covering ongoing maintenance

## Bounded Contexts

| Context | Responsibility | Key domain objects |
| --- | --- | --- |
| Flow Authoring | The canvas the maker draws on; ReactFlow nodes; Yjs document of the flow | Flow, Node, Edge, Host Adapter, Component Catalog (UI side) |
| Flow Runtime | Rust-side execution of the authored flow inside Tauri | FlowRuntime, Component (trait), ComponentRegistry, Wiring, Internal Event |
| Hardware IO | Single-owner serial connection to the Firmata board | BoardHandle, Board IO Loop, BoardConnection, BoardCommand, Hardware Callback |
| Firmware Flashing | Putting Firmata firmware onto a board from the desktop app | Flasher |
| Design-Tool Bridge | Figma and Penpot plugins that push design state into a flow via MQTT | `@microflow/mqtt`, Microflow Hardware Bridge plugin |
| LLM Integration | Pluggable LLM providers consumed by the `Llm` flow component | LLM Provider, Runtime Context |
| Collaboration | Real-time multi-user editing and persistence of flows | Yjs Doc, `@microflow/collab`, collab server |
| Identity & Billing | Accounts and subscriptions | better-auth, Polar, `@microflow/auth` |

## Out of Scope

- Writing raw C++ / Arduino firmware inside the editor — Microflow drives a Firmata-flashed board; firmware itself is fixed
- Browser-only / cloud editor — desktop-native via Tauri, because the core loop needs serial-port access
- Native non-Firmata protocols (CircuitPython, ESP-IDF, bare-metal) — out for now; could expand later

# ADR-0012 — Component trait plumbing stays explicit (no derive macro)

- **Status:** accepted (2026-06-26) — records a deliberate non-change
- **Date:** 2026-06-26
- **Deciders:** sander

> **Decision: keep the per-impl `Component` plumbing hand-written.** The trivial,
> uniform methods every node repeats — `base()`, `base_mut()`, `component_type()`,
> and the one-line `ComponentBuilder::build()` — stay explicit rather than being
> generated by a `#[derive(Component)]` proc-macro. An architecture review proposed
> the derive; **declined for now**, with a documented trigger for when to revisit.

## Context

Every one of the ~32 `Component` impls repeats the same shell: `base()` /
`base_mut()` returning `&self.base`, `component_type()` returning a string literal
equal to the struct name, and `impl ComponentBuilder { fn build … }` returning
`Ok(Self::new(id, config))`. The survey found this is **100% uniform** — same
`base: ComponentBase` field name, same `new(id, config)` signature, no exceptions —
so a `#[derive(Component)]` is mechanically feasible and would delete ~384 lines.

A review flagged it as a shallow-but-wide authoring surface and proposed the
derive. Two clarifications shaped the decision:

- It would **not** violate [ADR-0007](0007-node-wire-interface-emit-contract.md).
  That ADR rejected `macro_rules!` for *emit declaration* ("the crate today
  contains zero `macro_rules!`"); a derive proc-macro is a different mechanism, and
  the crate already runs on derives (`Serialize`/`Deserialize`/`TS`/`Error` ~150×).
  A derive that leaves `ports()`/`emits()`/`Config`/`dispatch` explicit honors
  ADR-0007. ADR-0007 line 115 even calls the duplication "macro-shareable later."
- So the real question is not rule-compliance but **leverage and legibility**.

## Decision

Keep it explicit. The deciding arguments:

1. **It deletes volume, not complexity.** Deletion test: remove `fn base() {
   &self.base }` and *zero* complexity reappears across callers — it never hid
   anything. Refactors earn their keep by concentrating complexity; there is none
   here. Low architectural leverage.
2. **A project-local derive is a private dialect.** `#[derive(Serialize)]` is
   understood by every Rust developer (and every code-reading tool) on sight; a
   `#[derive(Component)]` from a local proc-macro crate is vocabulary a reader must
   go learn by reading the macro's expansion. Explicit `fn base(&self) ->
   &ComponentBase { &self.base }` needs no lookup. This codebase optimizes for
   navigability (human and AI); the explicit form wins there.
3. **It preserves greppability.** `grep "fn component_type"` enumerates every
   node's type today; behind a derive it returns nothing.
4. **A 5th (proc-macro) crate is permanent surface** — `syn`/`quote` version
   churn, its own tests, and a single blast radius (a macro bug breaks *all* nodes
   at once). The crate's identity is lean and flat.
5. **The pain does not exist.** Nobody mistypes `&self.base` (it wouldn't compile),
   and the one drift that *would* be silent — ports/emits — is already guarded by
   `catalog_parity.rs`.

The full pros/cons of both sides were written up and weighed; this records the
outcome.

## Consequences

- ~384 lines of uniform, compile-checked plumbing remain. Accepted: it is
  consistent and legible, which reads as clean, not as debt.
- New nodes keep copying the shell. The cost is typing, not understanding.

## Revisit if (the trigger)

- Node authoring becomes a **measured** bottleneck (e.g. contributors cite the
  boilerplate as friction), **or**
- a real drift bug lands in the hand-written plumbing (a wrong `component_type`
  shipping).

When the trigger fires, the sanctioned escape is a **`#[derive(Component)]`
proc-macro** (not `macro_rules!`), leaving `ports()`/`emits()`/`Config`/`dispatch`
explicit. Opt-out per struct keeps any node needing a custom `build()`.

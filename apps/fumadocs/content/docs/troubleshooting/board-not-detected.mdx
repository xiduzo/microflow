---
title: Board not detected
description: Microflow Studio cannot find your Arduino. Things to check, in order.
---

If the connection panel sits at **Waiting**, fails at **Detecting**, or never advances past **Uploading**, work through these checks in order. The most common causes are at the top.

## 1. The cable

A surprising number of "broken Arduinos" are just bad cables. Many phone cables are **charge-only** and carry no data, even though they fit the port perfectly.

- Try a different USB cable. One that you have used to flash an Arduino with the official Arduino IDE before is ideal.
- Try a different USB port on your computer. USB hubs sometimes refuse to power a board reliably; plug straight into the computer.

## 2. The power LED on the board

When you plug the board in, a small power LED should light up (usually labelled `ON` or `PWR`).

- **No light** — the board is not getting power. Bad cable (see above) or bad port.
- **Light is on** — power is fine; move to the next check.

## 3. USB drivers

Some boards (especially clones) need a driver before the operating system will see them as a serial device.

- Open your operating system's device list:
  - **Windows** — *Device Manager → Ports (COM & LPT)*.
  - **macOS** — *System Information → USB*.
  - **Linux** — `ls /dev/tty*` before and after plugging in.
- If the board does not appear, install the right driver. See [Install board drivers](/docs/how-to/install-board-drivers).
- **Windows, CH340 clone:** if the board *does* appear as *USB-SERIAL CH340 (COM3)* but Microflow still fails with "device not functioning" (and it works fine on macOS), the newest CH340 driver (version 4.0) is rejecting the clone chip. Downgrade to CH340 driver version 3.4 — see [CH340 clones and the "device not functioning" error](/docs/how-to/install-board-drivers#ch340-clones-and-the-device-not-functioning-error-windows).

## 4. Another program is using the serial port

Only one program can hold a serial port at a time. The most common conflict is the Arduino IDE's **Serial Monitor**, but anything that connects to the board (PlatformIO, mosquitto pubsub examples, screen sessions) blocks Microflow.

- Close the Arduino IDE entirely (the Serial Monitor closing is not enough on Windows).
- Close any other tool talking to the board.
- Restart Microflow Studio.

## 5. Firmware corruption

If a different program has uploaded a custom sketch to the board, StandardFirmata is gone and Microflow's "Detecting" probes get no useful response.

- Open the connection panel in Studio and choose **Re-flash firmware** to reinstall StandardFirmata.
- See [Flash firmware manually](/docs/how-to/flash-firmware-manually) for a step-by-step.

## 6. The board itself

If everything else is ruled out, the board may be physically faulty.

- Open the Arduino IDE. Try the **Bare Minimum** example: *File → Examples → 01.Basics → BareMinimum*. Upload it. If even that fails with errors like "avrdude: stk500_recv()", the board has a bootloader issue and needs hardware-level recovery.
- Try a known-working board on the same computer. If it works, the original board is the problem.

## 7. You are running Microflow in a browser (Web Serial)

Microflow Studio can talk to your board in two ways: through **Web Serial** when you open the app in a Chromium-based browser, or through native USB when you use the **desktop app**. They behave differently, and a lot of "the board just won't connect" reports in the browser are actually Web Serial limitations rather than hardware faults.

### What Web Serial cannot do

- **Firefox and Safari** do not implement Web Serial at all. The sidebar will tell you to switch browsers or install the desktop app — there is no workaround inside those browsers.
- **The browser never auto-detects a board.** You have to click *Connect board* and pick the port from the browser's native chooser every session. Studio cannot remember the port across reloads on most platforms.
- **The browser cannot install USB drivers.** If your OS does not see the board as a serial device (see [USB drivers](#3-usb-drivers)), no browser can fix that.
- **The browser cannot flash a board that has no bootloader.** Web Serial can re-flash StandardFirmata on top of an existing bootloader, but a bricked board still needs the desktop app or the Arduino IDE.
- **Permission is per-origin, per-device.** If you previously granted access on `localhost` and are now on `app.microflow.tech` (or vice versa), you will be asked to authorise the board again.
- **Only one tab can hold the port.** If you have Microflow open in two tabs, the second one will fail to connect. Close the other tab first.
- **Chrome flags can disable it.** If Web Serial has been turned off in `chrome://flags` or by an enterprise policy, Studio will report the browser as unsupported.

### When the desktop app solves it

The desktop app (Tauri) talks to the serial port through the operating system directly and sidesteps most of the above. Prefer the desktop app when:

- You are on Firefox or Safari and cannot switch browsers.
- The board flashes fine from the Arduino IDE but not from the browser.
- You want the board to reconnect automatically after unplugging/re-plugging or reloading.
- You need to flash a board where the bootloader is intact but Web Serial refuses the write (some CH340 clones behave this way).
- You are on a corporate machine where Chrome policies block Web Serial.

The desktop app cannot fix a bad cable, a broken bootloader, or a missing OS-level driver — for those, keep working through the checks above.

Download the desktop app from the [Microflow releases page](https://github.com/xiduzo/microflow-t-stack/releases). Your flows are stored in the cloud (or locally, depending on your setup), so switching between browser and desktop does not lose any work.

## 8. Unsupported board

Microflow officially supports Arduino Uno, Mega, Leonardo, Micro, Nano, and Yun. Other boards may work but are not guaranteed. Boards based on ESP32, RP2040, Teensy, and similar typically need a different firmware path that Microflow does not handle today.

## When to ask for help

After working through the above, open an issue on the [GitHub repository](https://github.com/xiduzo/microflow-t-stack/issues) and include:

- Board name and source (genuine Arduino vs. clone)
- Operating system and version
- Microflow Studio version
- What appears in the connection panel
- Whether the Arduino IDE can talk to the same board
